Language design

Ownership and cleanup

Clyth 0.5.0 establishes explicit cleanup conventions for types that expose a real ownership contract. Automatic ownership inference remains future work.

Current rule: call .free() only when a type or module explicitly documents that method and transfers ownership to the caller. Do not guess based on an internal pointer or native implementation.

Borrowed opaque resources

Handle<T> represents a resource reference whose lifetime belongs to the runtime or native provider. A borrowed handle is not manually freed by ordinary application code.

struct Request {
    private Handle<RequestState> state
}

Explicitly owned resources

OwnedHandle<T> communicates that a wrapper owns the underlying resource. Its wrapper must provide a documented release path, conventionally .free().

struct NativeBuffer {
    private OwnedHandle<BufferState> state
}

void NativeBuffer.free() {
    native_buffer_release(this.state)
}

File I/O does not expose a file handle

The current read_file and write_file functions complete their operations behind the runtime boundary. User code receives a value or completion result, not a file pointer that must be released.

string text = read_file("config.json")
write_file("copy.json", text)

Future automatic cleanup

Scope-generated cleanup, string lifetime rules, estate inference, and MECC-driven collapse are planned for the self-hosted compiler and later alpha milestones. They are not silently performed by every 0.5.0 type.
