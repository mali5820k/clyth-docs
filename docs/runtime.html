Runtime Architecture

Clyth runtime modules are built around a deliberate boundary: Clyth source presents ergonomic types and methods; native code owns implementation details; the ABI between them stays small, explicit, and auditable.

The wrapper → ABI → provider chain

A native-backed module is not a random collection of extern functions. It is a layered design. The Clyth wrapper defines what application code sees. The ABI layer defines the stable functions and value representations crossing into native code. The provider implements those functions using a library such as libuv or yyjson.

Clyth application
    ↓ calls
Clyth wrapper type / method
    ↓ calls
extern C ABI function
    ↓ implemented by
native provider
    ↓ uses
libuv / yyjson / rapidhash / OS

Why handles exist

Native libraries often expose structs whose layout is not part of Clyth's language design. Passing those structs directly into Clyth would couple user code to native layout, native versioning, and implementation choices. A handle avoids that coupling.

Handle-based Request example

The Clyth Request value stores an opaque handle. The actual request state may contain parser buffers, socket state, libuv handles, body storage, and routing metadata. That state is owned by the native provider. Clyth methods ask the provider for specific information through ABI calls.

struct Request {
    private int64 native_handle
}

string Request.body() {
    return clyth_https_request_body(this.native_handle)
}

string Request.method() {
    return clyth_https_request_method(this.native_handle)
}

string Request.path() {
    return clyth_https_request_path(this.native_handle)
}

How a handle reaches the native function

  1. Native request arrives. libuv accepts the connection and llhttp parses the request line, headers, and body.
  2. Runtime state is stored. The native provider records request data in a table or connection-owned structure and assigns a handle.
  3. Clyth wrapper is created. The route callback receives a Request whose private field contains that handle.
  4. User calls a method. request.body() is a Clyth method, not direct native struct access.
  5. The method calls the ABI. The wrapper calls clyth_https_request_body(handle).
  6. The native provider resolves the handle. The C implementation looks up the request state and returns a Clyth string value.

ABI header and provider implementation

The wrapper knows the ABI function name because the module author declares it. The compiler links the native implementation because module metadata tells the build which native archive belongs to the wrapper. This is the answer to “where does request_body() come from?”: it is part of the module ABI contract.

// https.clyth wrapper layer
extern C string clyth_https_request_body(int64 request_handle)

string Request.body() {
    return clyth_https_request_body(this.native_handle)
}

// native/clyth_https.h
clyth_string clyth_https_request_body(int64_t request_handle);

// native/clyth_https.c
clyth_string clyth_https_request_body(int64_t request_handle) {
    request_state* request = lookup_request(request_handle);
    return clyth_string_from_bytes(request->body, request->body_length);
}

Why not expose raw native structs?

Exposing raw native structs makes a Clyth module fragile. If a public Clyth struct contains a native library layout, then upgrading the native library can become a language-level breaking change. It also makes it harder to replace a provider. A Request should still be a Request if the server internals evolve from libuv to another event provider later.

Design rule

The handle is not the public API. The Clyth methods are the public API. The handle is the bridge that lets the method reach runtime-owned state without exposing native memory layout.

Stable ABI shape

ABI functions should prefer stable Clyth-compatible values: integers, booleans, strings represented through the Clyth string ABI, handles, and simple structs whose layout is explicitly owned by Clyth. Avoid passing third-party native structs across the public module boundary.

Good ABI surface:
    int64 handle
    string clyth_https_request_body(int64 handle)
    void clyth_https_response_send(int64 handle, int32 status, string payload)

Fragile ABI surface:
    uv_stream_t* stream
    llhttp_t* parser
    OpenSSL SSL* object exposed to user code
