Module System

A Clyth module is the unit that connects user-facing source code to optional native implementation. The wrapper gives developers a Clyth API. The native binding supplies the implementation. Metadata tells the compiler and build scripts how to link the pieces together.

Module layout

The current repository keeps runtime material under clyth-runtime/. The exact packaging will continue improving, but the architecture is already visible: wrapper modules live as Clyth source, native bindings live in C or C-compatible code, and module metadata describes how to build and link them.

clyth-runtime/
├── modules/
│   ├── module-file-io/
│   │   ├── file-io.clyth          # Clyth wrapper API
│   │   └── module.json            # module metadata
│   ├── module-json/
│   │   ├── json.clyth
│   │   └── module.json
│   ├── module-hash/
│   │   ├── hash.clyth
│   │   └── module.json
│   └── module-https/
│       ├── https.clyth
│       └── module.json
│
├── c-bindings/
│   ├── file-io/                   # libuv-backed implementation
│   ├── http/                      # libuv + llhttp server core
│   ├── json/                      # yyjson bridge
│   └── hash/                      # rapidhash bridge
│
└── runtime_libraries.json          # runtime library discovery / linking map

Wrapper file responsibilities

The wrapper is what Clyth users read and import. It declares Clyth structs, methods, and extern ABI calls. It should hide provider details and expose names that make sense inside Clyth.

include "https"

struct Response {
    private int64 native_handle
}

void Response.send(int32 status, string payload) {
    clyth_https_response_send(this.native_handle, status, payload)
}

Native binding responsibilities

The native provider owns integration with C libraries. For 0.5.0 this includes libuv for event-loop and file I/O behavior, llhttp for HTTP parsing, yyjson for JSON, and rapidhash for hashing. The provider implements the ABI declared by the wrapper.

// native implementation sketch
void clyth_https_response_send(
    int64_t response_handle,
    int32_t status,
    clyth_string payload
) {
    response_state* response = lookup_response(response_handle);
    write_http_response(response, status, payload.data, payload.length);
}

module.json

module.json describes how a module is built and linked. It is the bridge between the source-level include statement and the native archives that must be passed into the final link command.

Metadata purpose

The compiler does not need to know what libuv is. It needs to know that including https requires the Clyth wrapper and the native archive produced for that module. Metadata keeps that knowledge outside the language grammar.

{
  "name": "https",
  "kind": "native-backed-module",
  "wrapper": "https.clyth",
  "build": {
    "sources": [
      "../../c-bindings/http/http.c",
      "../../c-bindings/libuv/vendor/libuv/src/unix/core.c"
    ],
    "include_dirs": [
      "../../c-bindings/http/include",
      "../../c-bindings/libuv/vendor/libuv/include"
    ],
    "c_flags": ["-D_GNU_SOURCE", "-D_FILE_OFFSET_BITS=64"],
    "output": "libclyth_https.a"
  },
  "licenses": ["MIT", "BSD-2-Clause"]
}

How the compiler knows what to bundle

  1. The source file says include "https".
  2. The compiler resolves the module name against runtime module metadata.
  3. The wrapper source contributes declarations and Clyth methods.
  4. The module's native archive is added to the link command.
  5. The final executable contains the user's generated code plus the required runtime module code.

Creating a custom native-backed module

Example module skeleton

A new module should expose Clyth-friendly types and keep native structs under the provider. Handles are the common pattern when native state needs to survive across calls.

module-image/
├── image.clyth
├── module.json
└── native/
    ├── image_abi.h
    └── image_abi.c

Custom wrapper example

The Clyth wrapper declares the ABI calls and then wraps them in methods. Users import image and work with Image, not with the decoder library internals.

extern C int64 clyth_image_open(string path)
extern C int32 clyth_image_width(int64 handle)
extern C int32 clyth_image_height(int64 handle)
extern C void clyth_image_close(int64 handle)

struct Image {
    private int64 handle

    Image(string path) {
        this.handle = clyth_image_open(path)
    }
}

int32 Image.width() {
    return clyth_image_width(this.handle)
}

int32 Image.height() {
    return clyth_image_height(this.handle)
}

void Image.close() {
    clyth_image_close(this.handle)
}

Licensing expectations

Clyth prefers permissively licensed dependencies for statically distributed artifacts: MIT, BSD-2-Clause, BSD-3-Clause, Zlib, and Apache-2.0 when explicitly tracked. GPL, LGPL, and AGPL dependencies complicate static distribution and should be avoided unless a future release deliberately documents that tradeoff.
