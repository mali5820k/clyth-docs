Language Guide

This guide documents the 0.5.0 language surface: variables, arrays, structs, methods, extension methods, functions, function values, lambdas, generics, and template interpolation. It is written as an engineering reference rather than a syntax teaser.

Variables and valuesPrimitive declarations, strings, arrays, and initialization. Structs and methodsPublic-by-default fields, private fields, constructors, explicit this, and extension methods. Functions and callbacksFunction values, function<Return(Args...)>, lambdas, and inferred void callbacks.

Variables and primitive values

Clyth declarations place the type first, followed by the variable name. The current alpha subset focuses on explicit declarations because the compiler is still stabilizing semantic analysis and lowering. The intent is to keep systems code readable at the point where storage and type decisions are made.

int32 count = 4
bool enabled = true
string label = "runtime-foundation"

count = count + 1

if enabled {
    println(`label: ${label}`)
}

Arrays and indexed values

Fixed arrays are part of the language surface. They are useful where the length belongs to the declaration rather than to a heap container. Dynamic collections such as List<T> belong to the runtime collection module.

int32[4] ports = [8080, 8081, 9090, 9443]

int32 first = ports[0]
ports[1] = 7070

for_index = 0
while for_index < 4 {
    println(`${ports[for_index]}`)
    for_index = for_index + 1
}

Struct layout and visibility

Struct fields are public by default. This keeps simple data types concise. Use private for fields that represent implementation details, especially native handles or invariants that should only be touched through methods.

struct Request {
    private int64 native_handle
}

struct Point2D {
    float x
    float y
}

Point2D point = Point2D(3.0, 4.0)
println(`${point.x}`)

Constructors and explicit this

Constructors initialize struct values. Methods use explicit this when accessing the receiver. That explicitness matters for later ownership and lowering work: the compiler does not have to guess whether a name refers to a local value or a field.

struct Counter {
    int32 value

    Counter(int32 initial) {
        this.value = initial
    }

    void increment() {
        this.value = this.value + 1
    }
}

Counter counter = Counter(10)
counter.increment()

Extension-style methods

Methods can also be declared outside the struct body. This separates data layout from implementation. The rule is additive: an extension method may add behavior for a type, but duplicate definitions for the same method signature should be rejected so imports do not silently redefine behavior.

struct Vector2 {
    float x
    float y
}

float Vector2.length_squared() {
    return this.x * this.x + this.y * this.y
}

Vector2 v = Vector2(3.0, 4.0)
println(`${v.length_squared()}`)

Functions and return values

Functions are declared with an explicit return type. A function returning void performs work without producing a value. Non-void functions must return a value compatible with their declared type.

int32 add(int32 left, int32 right) {
    return left + right
}

void log_sum(int32 left, int32 right) {
    println(`${left + right}`)
}

First-class function values

The canonical callable type is function<Return(Args...)>. The angle brackets bind the type, while the parentheses read like a callable signature. This syntax is easier to scan than nesting a return type and a separate argument list as unrelated generic parameters.

int32 run_twice(function<int32(int32)> callback, int32 value) {
    return callback(callback(value))
}

int32 double_value(int32 value) {
    return value * 2
}

int32 result = run_twice(double_value, 5)

Anonymous functions and lambdas

Lambdas are used heavily by the server runtime because route handlers are callbacks. When no return value is produced, the lambda is treated as void. When return statements exist, the compiler should infer a consistent return type or report a mismatch.

router.post("/api/message", (Request request, Response response) => {
    string body = request.body()
    println(`received: ${body}`)
    response.send(200, "OK")
})

function<int32(int32, int32)> add = (left, right) => {
    return left + right
}

Generics and runtime collections

Generic syntax is used for typed runtime containers and function values. Collections such as List<T>, Set<T>, and Map<K,V> are runtime library types rather than compiler magic. That keeps collection behavior documented and replaceable while preserving typed usage in Clyth source.

include "collections"

List<int32> values = List<int32>()
values.push(10)
values.push(20)

Map<string, int32> scores = Map<string, int32>()
scores.set("compiler", 1)
scores.set("runtime", 2)

Template interpolation

Backtick template strings evaluate expressions inside ${...}. Interpolation should not be limited to simple identifiers; method calls, arithmetic, and stringify calls are part of the expected language behavior.

string user = "Clyth"
int32 major = 0
int32 minor = 5

println(`hello ${user}`)
println(`release ${major}.${minor}`)
println(`payload: ${JSON.stringify(payload)}`)
println(`sum: ${10 + 20}`)
