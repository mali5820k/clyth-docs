Tooling and Build

The 0.5.0 build is still a bootstrap toolchain, but its release policy is now clear: native artifacts should be lean, auditable, and built through the intended musl/libc++ path.

Build the compiler

The main build script prepares dependencies, builds the compiler, builds runtime native archives, and optionally packages a distribution.

./build_compiler.sh

# Faster development build without creating dist artifacts
./build_compiler.sh --skip-dist

Compile Clyth source

The compiler currently lowers Clyth to LLVM IR, then links a native executable using the configured toolchain path.

./build-compiler/clyth_compiler_bin     -c sample-clyth-programs/web-app-preview/clyth_backend_welcome.clyth     -o sample-clyth-programs/.sample-bin/clyth_backend_welcome

Regression suite

The regression suite compiles numbered sample programs and executes them. A sample passes when it exits with code 0. This avoids brittle stdout comparisons and lets each sample validate its own behavior.

./run_regression_tests.sh

ABI audit

The ABI audit checks release artifacts for accidental GNU/glibc/libstdc++ leakage. It is intentionally focused on actual dependency evidence rather than strings that linkers may contain merely because they understand other ABIs.

./audit_no_gnu_abi.sh

# Expected release-candidate result:
# [ OK ] No GNU/glibc/libstdc++ ABI markers found in checked release artifacts.

Distribution policy

Distribution artifacts should include the compiler binary, runtime files, samples, documentation, inspection tools, and license files. They should not include LLVM source trees, LLVM build directories, Zig toolchains, generated object caches, or unrelated vendor checkouts.

Why LLVM is still present

LLVM remains valuable as the backend pipeline: optimization, object generation, target support, and linker ecosystem. The 0.6.0 direction is not to reject LLVM, but to reduce the compiler's reliance on LLVM's C++ APIs by generating LLVM IR from Clyth-owned compiler structures.
