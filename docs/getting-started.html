Getting Started

Clyth projects are compiled ahead of time. Today the compiler emits LLVM IR and links a native executable. User code imports runtime capability through Clyth modules such as file-io, json, and https.

A minimal program

The standard output path is print and println. Direct extern C printf declarations remain useful for interop experiments, but ordinary Clyth programs should use the runtime I/O API.

include "file-io"

int32 main(string[] args) {
    println("Hello from Clyth")
    return 0
}

Compile a sample

The compiler accepts a Clyth source file and writes a native executable. Output directories are created automatically by the current compiler path.

./build-compiler/clyth_compiler_bin     -c sample-clyth-programs/01_hello.clyth     -o sample-clyth-programs/.sample-bin/hello

./sample-clyth-programs/.sample-bin/hello

Include runtime modules

include pulls a Clyth wrapper module into the program. The compiler and runtime metadata decide which native archive must be linked for that wrapper. The source file imports the API; the build system links the backing implementation.

include "file-io"
include "json"
include "https"

int32 main(string[] args) {
    string payload = JSON.stringify({ "ready": true })
    write_file("status.json", payload)
    println(read_file("status.json"))
    return 0
}

Release validation commands

./build_compiler.sh --skip-dist
./run_regression_tests.sh
./audit_no_gnu_abi.sh

The regression suite checks language and runtime behavior. The ABI audit checks built artifacts for accidental GNU/glibc/libstdc++ leakage. Both are release gates for the 0.5.0 direction.
