Design principles

Clyth is being designed around a readable compiler, explicit native boundaries, deterministic behavior, and a path from a practical alpha foundation to a self-hosted 1.0 language.

Readable systems code

Language and compiler features should be understandable without requiring contributors to decode a monolithic frontend. The self-hosted compiler is therefore planned as small, composable phases with narrow interfaces.

source
  → tokens
  → syntax tree
  → semantic model
  → Clyth IR
  → LLVM IR
  → native program

Explicit runtime boundaries

Clyth-facing APIs wrap stable C ABI functions. Native implementation details remain behind that boundary so runtime providers can change without forcing application code to depend on C or C++ layouts.

Clyth method
  → extern C ABI function
  → native provider
  → operating system

No raw-pointer language surface

Native providers may use pointers internally, but user-facing Clyth APIs should expose arrays, values, collections, or opaque resource wrappers with an explicit ownership contract. This keeps pointer arithmetic and provider-specific layouts outside ordinary application code.

struct Request {
    private Handle<RequestState> state
}

Foundation before permanence

The 0.5.0 compiler exists to prove the pipeline and build the next compiler. Features that would complicate this transitional frontend may be deferred so their first durable implementation can be written in Clyth.

0.5.0 C++ compiler
        ↓
0.6.0 Clyth compiler
        ↓
self-rebuild and expansion
        ↓
1.0 stable language

MECC as the ownership destination

Managed Entanglement for Collapsible Collections organizes ownership around related estates. A root estate losing its final relationship can cause known dependent estates to collapse through compiler-visible cleanup paths. This is a long-term design goal, not a claim about the released 0.5.0 runtime.

root estate reaches zero
  → release owned resources
  → decrement dependent estates
  → collapse dependents at zero
