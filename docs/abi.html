ABI Boundary

Clyth modules use a wrapper → ABI → native-provider model. Clyth owns the user-facing object model. Native libraries provide capability behind a small set of exported functions.

Call path from method to native provider

When code calls request.body(), it is not reading fields from a native HTTP parser. It is invoking a Clyth method that passes an opaque resource to a stable ABI function.

router.post("/", (Request request, Response response) => {
    string body = request.body()
    response.send(200, body)
})

// request.body()
//   -> Request.body(request)
//   -> https_request_body(request.native_handle)
//   -> C ABI
//   -> native runtime lookup

Where ABI functions come from

The module wrapper declares which native functions exist. The C binding implements those functions and owns the tables, handles, sockets, parser states, and native allocations.

// https.clyth
extern C string https_request_body(Handle<RequestState> handle)

struct Request {
    private Handle<RequestState> native_handle
}

string Request.body() {
    return https_request_body(this.native_handle)
}

Module layout

A native-backed module has a Clyth wrapper, module metadata, and native implementation files. The wrapper exposes the language API; the native side does the system work.

module-https/
├── module.json
├── https.clyth
└── native/
    ├── https_abi.h
    └── https_abi.c

c-bindings/http/
├── http.c
├── http.h
└── vendor/libuv, llhttp
