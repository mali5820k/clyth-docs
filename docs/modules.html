Modules

A module is the unit that connects Clyth source wrappers, native ABI bindings, metadata, and license obligations.

module.json

module.json describes the module name, source files, native archives, and dependency surface. The compiler/build layer uses this metadata to decide which wrappers and archives to include.

{
  "name": "module-json",
  "sources": ["json.clyth"],
  "native_archives": ["x86_64/libclyth_json.a"],
  "license": "MIT"
}

Wrapper layer

The wrapper layer is ordinary Clyth source. It defines structs, methods, and functions that application code uses.

extern C JsonDocument json_parse(string text)

struct JSON {}

JsonDocument JSON.parse(string text) {
    return json_parse(text)
}

Native provider

The native provider implements the ABI functions and owns native layout. It may call libuv, yyjson, rapidhash, OpenSSL, or another permissively licensed library.

// json_bindings.c
JsonDocument json_parse(clyth_string text) {
    // yyjson implementation details remain here
}

Writing your own module

Start with a Clyth wrapper and a narrow ABI. Expose Clyth objects and methods, not native memory layouts. Use Handle<T> or OwnedHandle<T> when native state must cross the boundary.

module-image/
├── module.json
├── image.clyth
└── native/
    ├── image_abi.h
    └── image_abi.c
