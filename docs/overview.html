Overview

Clyth 0.5.0 is the Runtime Ecosystem Foundation release. The point of this milestone is not only that Clyth can emit native executables; it is that Clyth code can compile, cross a stable ABI boundary, and drive native-backed runtime capabilities while preserving a language-level API surface.

Language surfaceStructs, methods, extension methods, function values, lambdas, arrays, strings, templates, and generics. Runtime architectureHow Clyth wrapper methods map onto ABI functions and native providers. Module developmentHow module.json, Clyth wrappers, and native source files compose into a linkable module.

What 0.5.0 establishes

Clyth is currently an ahead-of-time compiler that lowers the supported alpha language subset through LLVM IR and links native Linux artifacts through a Zig/musl toolchain path. The 0.5.0 release candidate validates the language/runtime relationship with a browser-facing server demo, JSON request handling, libuv-backed file I/O, first-class callbacks, and recursive ABI auditing.

The important design result is a repeatable runtime pattern: Clyth owns the public API, the ABI boundary owns stability, and the native provider owns implementation detail.

0.5.0 validated capabilities

LanguageStructs, constructors, explicit this, extension methods, function values, lambdas, template interpolation, native strings, arrays, and generic runtime collections.
Runtimeprint/println, libuv-backed file I/O, yyjson-backed JSON, rapidhash hashing, and a unified server abstraction using Request and Response wrappers.
BuildLLVM IR emission, Zig/musl link path, LLVM libc++ C++ ABI, and audit checks for GNU/glibc/libstdc++ leakage.
TestingThe release candidate passes 20/20 regression samples, including web callbacks, detached methods, file I/O, function values, and runtime modules.

What belongs in Clyth, and what belongs below it

Clyth should not expose every native library structure directly. A systems language still needs good boundaries. A native provider such as libuv or yyjson can be excellent at implementation while still being the wrong shape for the public language API. Clyth modules therefore wrap native capability in Clyth-facing types and methods.

That boundary makes future changes possible: the HTTP stack can evolve, file I/O can gain asynchronous variants, JSON internals can be replaced or expanded, and the compiler can move toward a bootstrapped frontend without forcing user code to track native layout changes.
